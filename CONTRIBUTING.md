# Contribute to Snapfig

먼저, 기여해 주셔서 감사합니다! ❤️

모든 형태의 기여는 환영하며 소중히 여깁니다. 다양한 기여 방식과 프로젝트에서 이를 어떻게 다루는지에 대한 자세한 내용은 [목차](#table-of-contents)를 참고해 주세요. 기여하시기 전에 해당되는 섹션을 반드시 읽어주세요. 저희 유지관리자들에게도 많은 도움이 되고, 참여하는 모든 분들에게 보다 원활한 경험을 제공할 수 있습니다. 커뮤니티는 여러분의 기여를 기다리고 있습니다. 🎉

> 그리고 프로젝트가 마음에 들지만 기여할 시간이 없으시더라도 괜찮습니다. 프로젝트를 응원하고 감사를 표현할 수 있는 쉬운 방법들이 있으니, 아래의 방법들도 환영합니다:
> - 프로젝트에 별(Star) 표시하기
> - 트윗으로 프로젝트 소개하기
> - 자신의 프로젝트 README에 본 프로젝트 언급하기
> - 지역 밋업에서 언급하거나 친구/동료에게 소개하기

## 행동 강령

이 프로젝트와 그에 참여하는 모든 사람은 [Snapfig 행동 강령](https://github.com/CommitBang/Backend/blob//CODE_OF_CONDUCT.md)에 의해 관리됩니다. 참여 시 해당 강령을 준수해야 합니다.

## 질문이 있어요

> 질문하기 전에, [문서](README)를 먼저 읽었는지 확인해 주세요.

질문하기 전에는 기존의 [이슈](https://github.com/CommitBang/Backend/issues) 중 관련된 것이 있는지 검색해 보세요. 적절한 이슈를 찾았지만 추가 설명이 필요하다면, 해당 이슈에 질문을 남기면 됩니다. 인터넷 검색도 먼저 해보는 것이 좋습니다.

그럼에도 불구하고 질문이 필요하다면 아래의 방법을 추천합니다:

- 새로운 [이슈](https://github.com/CommitBang/Backend/issues/new)를 생성합니다.
- 문제가 발생한 상황에 대해 가능한 한 많은 정보를 제공합니다.

그 후, 저희가 최대한 신속하게 대응하겠습니다.

## 기여하고 싶어요


> ### 법적 고지 <!-- omit in toc -->
> 본 프로젝트에 기여하는 경우, 본인이 100% 직접 작성한 콘텐츠이며 해당 콘텐츠에 대한 권리를 보유하고 있으며, 프로젝트 라이선스 하에 제공되는 것에 동의해야 합니다.

### 기여 방법
- 버그 리포트: 발견한 문제점 신고
-  기능 개선 제안: 새로운 기능 아이디어 제안
-  코드 기여: 버그 수정, 기능 구현, 성능 개선
-  문서 개선: README, 가이드, 주석 개선
-  테스트 추가: 테스트 커버리지 향상
-  UI/UX 개선: 사용자 경험 향상

### 버그 제보하기

<!-- omit in toc -->
#### 버그 리포트 제출 전

좋은 버그 리포트는 다른 사람들이 추가 정보를 물어보지 않도록 충분히 자세해야 합니다. 따라서 다음의 단계를 사전에 완료해 주세요.

- 최신 버전을 사용 중인지 확인합니다.
- 이것이 진짜 버그인지, 혹은 환경 설정이나 버전 호환 문제 등 본인의 실수인지 확인합니다. ([문서]()를 반드시 읽고, [질문 섹션](#i-have-a-question)도 확인해 보세요.)
- 동일한 문제가 이미 [버그 트래커](https://github.com/CommitBang/Backend/issues?q=label%3Abug)에 등록되어 있는지 확인합니다.
- 인터넷(예: Stack Overflow)에서도 유사한 문제가 논의되었는지 검색해 봅니다.
- 다음 정보를 수집해 주세요:
  - 에러 스택 트레이스
  - OS, 플랫폼, 아키텍처 정보 (Windows, Linux, macOS, x86, ARM 등)
  - 관련 환경 버전 (인터프리터, 컴파일러, SDK, 런타임 등)
  - 입력값과 출력값
  - 재현 가능한지, 구 버전에서도 동일한 문제가 발생하는지

<!-- omit in toc -->
#### 좋은 버그 리포트 작성 방법

> 보안 관련 이슈나 민감한 정보가 포함된 버그는 절대 공개 이슈 트래커나 공공 장소에 보고하지 마세요. 대신 이메일 <>로 보내주세요.

버그 및 에러는 GitHub 이슈를 통해 관리됩니다. 문제가 생겼다면:

- [이슈](https://github.com/CommitBang/Backend/issues/new)를 엽니다. (아직 진짜 버그인지 확실하지 않기 때문에, 이슈 제목에 'bug'를 쓰지 말고 라벨도 붙이지 마세요.)
- 기대하는 동작과 실제 동작을 설명하세요.
- 재현 단계를 포함한 최대한의 정보를 작성하세요. 가능하다면 최소한의 코드로 문제를 재현해 주세요.
- 앞에서 수집한 정보를 첨부해 주세요.

이슈가 등록되면:

- 프로젝트 팀이 적절한 라벨을 지정합니다.
- 팀원이 제공된 단계로 재현을 시도합니다. 재현이 안 될 경우, 추가 정보를 요청하며 `needs-repro` 라벨을 붙입니다.
- 재현에 성공하면 `needs-fix` 및 다른 관련 라벨을 붙이고, [기여자](#your-first-code-contribution)가 해결할 수 있도록 열어둡니다.

### 기능 개선 제안하기

이 섹션은 Snapfig에 새로운 기능이나 기존 기능 개선을 제안하는 방법을 안내합니다. 아래 지침을 따르면 커뮤니티와 유지관리자가 여러분의 제안을 잘 이해할 수 있습니다.

<!-- omit in toc -->
#### 제안하기 전에

- 최신 버전을 사용 중인지 확인합니다.
- [문서]()를 읽고, 이미 설정으로 구현 가능한 기능이 아닌지 확인합니다.
- [이슈 검색](https://github.com/CommitBang/Backend/issues)을 통해 기존 제안이 있는지 확인하고, 있다면 새 이슈를 만들지 말고 댓글을 답니다.
- 제안이 본 프로젝트의 목표와 맞는지 고민합니다.

<!-- omit in toc -->
#### 좋은 제안 작성 방법

기능 개선 제안은 [GitHub 이슈](https://github.com/CommitBang/Backend/issues)를 통해 접수됩니다.

- 제안을 식별할 수 있는 **명확하고 간결한 제목**을 작성합니다.
- 제안 내용을 **단계별로 구체적으로 설명**합니다.
- 현재 동작과 기대하는 동작을 비교해 설명하고, 기존 방법이 왜 부족한지도 서술합니다.
- 가능하면 **스크린샷이나 화면 녹화**를 첨부합니다.
- 이 기능이 왜 유용한지, 다른 프로젝트는 어떻게 해결했는지 등을 언급해 주세요.

### 첫 코드 기여
Snapfig에 처음으로 코드 기여를 하고 싶다면 아래 절차를 따라주세요.
~~~
1. 저장소를 포크(Fork)하고 클론(clone)합니다.
2. 로컬 환경에서 서버를 실행해 확인합니다.
3. `feature/`, `bugfix/` 등 브랜치 네이밍 컨벤션에 따라 새 브랜치를 생성합니다.
4. 필요한 코드를 수정하고, 테스트를 작성합니다. (테스트가 필요한 경우)
5. 커밋 메시지 규칙에 맞게 커밋합니다. (아래 [커밋 메시지](#커밋-메시지) 참고)
6. 푸시 후, GitHub에서 Pull Request를 생성합니다.
7. 리뷰를 기다리고, 피드백이 있다면 반영합니다.
~~~
가능하면 작은 단위의 기여부터 시작해 주세요. 예: 잘못된 주석 수정, 간단한 리팩토링, 불필요한 콘솔 로그 제거 등.

### Pull Request 작성

Snapfig에 코드나 문서 등을 기여할 때는 반드시 Pull Request(PR)를 통해 제출해 주세요.

#### PR 작성 전 체크리스트

- [ ] 로컬 브랜치를 최신 main/master 브랜치에서 시작했나요?
- [ ] 기능/수정이 완료되었고, 관련 테스트도 통과했나요?
- [ ] 커밋 메시지가 [커밋 메시지 컨벤션](#커밋-메시지) 가이드를 따르나요?
- [ ] 문서가 필요한 경우, 문서도 함께 수정했나요?
- [ ] 변경 내용에 대한 간단한 설명이 PR에 포함되어 있나요?

#### PR 제목 및 본문 작성 규칙

- 제목은 **간결하고 명확하게** 변경 내용을 요약합니다.
  - 예: `feat: 사용자 로그인 페이지 구현`
- 본문에는 다음 내용을 포함하는 것이 좋습니다:
  - 변경한 이유
  - 구현한 방식
  - 관련된 이슈 번호 (예: `Closes #42`)
  - 참고 링크 및 스크린샷 (필요 시)

#### PR 병합 정책

- 최소 1명 이상의 리뷰어 승인을 받아야 병합됩니다.
- 팀의 병합 방식은 보통 다음 중 하나로 결정됩니다:
- 리뷰어가 요청한 변경 사항이 있다면 성실히 반영해 주세요.
- 모든 CI 검사(build, lint, test)가 통과해야 합니다.



### 코드 리뷰

Snapfig 프로젝트의 코드 리뷰는 다음과 같은 절차를 따릅니다:

#### 🔁 1. PR 생성
- 작성자는 `pull_request_template.md`에 따라 PR을 작성합니다.
- PR 생성 후, 리뷰어를 지정하거나 `@팀`을 태그하여 리뷰를 요청합니다.

#### 👀 2. 리뷰어 확인 및 리뷰 시작
- 리뷰어는 PR을 확인하고 다음을 점검합니다:
  - 기능 동작 확인
  - 코드 스타일, 네이밍 규칙 일관성
  - 비즈니스 로직의 타당성
  - 테스트 코드 유무 및 범위

#### 💬 3. 피드백 및 응답
- 리뷰어는 GitHub의 `Comment` 또는 `Suggestion` 기능을 통해 구체적이고 개선 가능한 피드백을 남깁니다.
- 작성자는 모든 피드백에 대해 `답변` 또는 `반영` 후, 변경 사항을 커밋합니다.
- 필요한 경우 `force push`가 아닌 `추가 커밋`을 선호합니다.

#### ✅ 4. 리뷰 승인 및 병합
- 모든 리뷰가 완료되면 `Approve` 후 병합합니다.
- CI(Build, Lint, Test)가 모두 통과해야 병합됩니다.

---

### 코드 리뷰 기준

리뷰는 다음 항목을 기준으로 수행합니다:

#### 1. 기능 정확성 (Correctness)
- 기능이 명세서나 이슈 내용대로 정확히 구현되었는가?
- 엣지 케이스를 고려했는가?

#### 2. 코드 품질 (Code Quality)
- 중복 코드는 없는가?
- 함수/모듈이 명확하게 분리되어 있는가?
- 변수명, 함수명이 명확하고 의미가 있는가?

#### 3. 테스트 (Test Coverage)
- 유닛/통합 테스트가 작성되어 있는가?
- 기존 테스트를 깨뜨리지 않았는가?

#### 4. 스타일 및 일관성 (Style & Consistency)
- 프로젝트의 코드 스타일 가이드를 따르고 있는가?

#### 5. 성능 및 보안 (Performance & Security)
- 성능상 병목 요소는 없는가?
- 민감 정보 노출이나 인증 관련 취약점은 없는가?

#### 6. 문서 및 주석 (Documentation & Comments)
- 변경된 기능이 README 또는 관련 문서에 반영되었는가?
- 필요한 경우 코드에 적절한 주석이 있는가?

## 스타일 가이드
### 브랜치 네이밍

브랜치는 작업 목적에 따라 접두어(prefix)를 사용하며, 브랜치명은 소문자와 하이픈(-)으로 구성합니다.

| 유형        | 접두어       | 예시                              | 설명                             |
|-------------|--------------|------------------------------------|----------------------------------|
| 기능 추가   | `feature/`   | `feature/[name]`              | 새로운 기능, UI, API 등         |
| 버그 수정   | `bugfix/`    | `bugfix/[name]`          | 버그 수정                        |
| 핫픽스       | `hotfix/`    | `hotfix/[name]`           | 긴급 수정 (배포 중단 방지 등)   |
| 리팩토링     | `refactor/`  | `refactor/[name]`       | 구조 변경, 동작 변화 없음       |
| 문서 수정   | `docs/`      | `docs/[name]`              | README, 문서 파일 수정          |
| 테스트       | `test/`      | `test/[name]`             | 테스트 코드 추가 및 수정        |
| 설정 변경   | `chore/`     | `chore/[name]`      | 빌드/배포 설정, 패키지 업데이트 |

### 커밋 메시지

```
<type>([optional scope]): <short summary>

[optional body]

[optional footer(s)]
```

#### 주요 타입

| 타입       | 설명                                   |
|------------|----------------------------------------|
| `feat`     | 새로운 기능                             |
| `fix`      | 버그 수정                               
| `docs`     | 문서 관련 변경                          |
| `style`    | 코드 스타일 수정 (기능 변경 없음)       |
| `refactor` | 코드 리팩토링 (기능 변경 없음)          |
| `test`     | 테스트 코드 추가/수정                   |
| `chore`    | 빌드, 설정, 패키지 등 기타 변경         |
